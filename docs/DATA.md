## Executive Summary

This document defines the data architecture that bridges the **Entity Data Editor (EDE)** frontend with the **Ontology YAML** backend storage system. The EDE provides a spreadsheet-style UI for IT administrators and organizational managers, while the Ontology system provides structured, validated, relationship-aware data persistence for corporate entities.

## 1. Concept Mapping

### 1.1 EDE â†” Ontology Terminology

| EDE Concept | Ontology Concept | Description |
|-------------|------------------|-------------|
| **Table** | `Class` | A type definition (e.g., Person, Team, Product) |
| **Row** | `ClassInstance` | A single entity with `_class` and `_id` |
| **Column Group** | `Component` | Reusable property groupings |
| **Column** | `Property` | Individual field within a component |
| **Cell** | Property value | The actual data for a property |
| **Entity Reference** | `Relation` | Links between instances |
| **Tier 1 (Activity)** | `namespace` | Logical grouping by team role |
| **Tier 2 (Class)** | `_class` | Filter by entity class type |
| **Tier 3 (Component)** | localName (component) | Filter by component (ie. edit across all matching Classes, which have this component) |

### 6.3 Query Syntax

The EDE search bar uses the existing `ontology search` DSL syntax. This supports two primary patterns plus boolean operators.

#### Query Grammar (EBNF)

```
query           = expression ;
expression      = term { OR term } ;
term            = factor { AND factor } ;
factor          = NOT factor | LPAREN expression RPAREN | match ;
match           = relation_match | class_match | bare_value ;

class_match     = [ id ] COLON [ class ] [ DOT property ] VALUE_SEP value
relation_match  = [ entity_spec ] REL_START [ COLON RELATION ] RBRACKET [ DOT qualifier ] ARROW [ VALUE_SEP value ]
entity_spec     = LPAREN [ id ] [ COLON class ] RPAREN
```

## 9. Activity Views

Activity views are UI-only configurations stored separately from the Ontology data. They define role-specific views of the data (e.g., HR, IT Ops, Product Manager).

### 9.1 View Configuration Files

View configs are plain YAML files stored in a `config/views/` directory, independent of the Ontology format:

```yaml
# config/views/hr.view.yaml
name: "ğŸ‘¥ HR Manager"
icon: "users"
color: "#4A90D9"

# Visible classes (Tier 2 tabs)
classes:
  - Person
  - Team

# Column presets per class
columns:
  Person:
    visible:
      - identity.name
      - identity.givenName
      - identity.surname
      - contact.email
      - employment.title
      - employment.department
      - employment.active
      - profile.external
    hidden:
      - employment.created
      - profile.nickname
  Team:
    visible:
      - naming.name
      - info.shorthand

# Default filters (uses ontology search syntax)
filters:
  Person: ":Person.employment.active: true"

# Sort preferences
sort:
  Person: { column: "identity.surname", direction: "asc" }
  Team: { column: "naming.name", direction: "asc" }
```

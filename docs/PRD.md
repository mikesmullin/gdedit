# Product Requirements Document: Game Data Editor

## Executive Summary

**Product Name:** Game Data Editor (GDE)  
**Version:** 1.0  
**Date:** February 5, 2026  
**Author:** Game Development Team

The Game Data Editor is a browser-based, spreadsheet-style tool for managing game entities, components, and assets. It combines the functionality of a Unity Hierarchy window, Inspector pane, and asset browser into a single, powerful data management interface. The tool is engine-agnostic and operates on flat-file formats (CSV, JSON, YAML) with hot-reload capability.

---

## 1. Problem Statement

Game developers face several pain points when managing game data:

1. **Tedious Entity Editing**: Editing multiple entities requires repetitive select â†’ copy â†’ paste workflows
2. **Data Fragmentation**: Game data is scattered across multiple files and formats
3. **Lack of Visibility**: Difficult to see relationships between entities and assets
4. **No Bulk Operations**: Hard to modify many entities at once
5. **Poor Collaboration**: Different team members (audio, art, design) need different views of the same data

---

## 2. Product Goals

### Primary Goals
- Provide a unified spreadsheet interface for all game data
- Support bulk editing operations across multiple entities
- Enable activity-based views for different team roles
- Maintain engine-agnostic flat-file compatibility
- Support hot-reload for rapid iteration

### Success Metrics
- 70% reduction in time spent on bulk entity modifications
- Support for projects with 10,000+ entities
- Sub-100ms hot-reload times
- Zero data loss during save/load operations

### 3.2 Key Principles

| Principle | Description |
|-----------|-------------|
| **1 Row = 1 Asset** | Each row represents either a file on disk OR a GameObject in scene |
| **1 Column = 1 Component Field** | Component fields are the fundamental column unit (Flecs-compatible) |
| **1 Table = 1 Class** | Each table corresponds to a `.c` or `.cs` class definition |
| **Flat-File First** | All data is serializable to CSV/JSON/YAML |

## 4. Feature Requirements

### 4.1 Tier System (Navigation Hierarchy)

| Tier | Name | Purpose | UI Element |
|------|------|---------|------------|
| **Tier 1** | Window | Activity-based views (Audio, Art, Design) | Top-level tabs |
| **Tier 2** | Tab | GameObject type filter (Prefab, Scene, Mesh) | Secondary tabs |
| **Tier 3** | Sub-Tab | Namespace/Class filter (RPG.Unit â†’ "Unit") | Tertiary tabs |
| **Tier 4** | Child-Tab | Nested hierarchy navigation | Quaternary tabs |

### 4.2 Cell Widgets

The editor must support custom widgets for the following data types:

**Primitive Types:**
- Integer, Byte, Short, Long, Unsigned Int
- Float, Double
- Bool (checkbox)
- String (text input)
- Enum (dropdown)

**Complex Types:**
- Color (color picker popup)
- Vector2/3/4 (multi-field input)
- Gradient (gradient editor popup)
- Curve (curve editor popup)
- Arrays/Lists (expandable or JSON view)
- Nested Objects (expandable or JSON view)

**Reference Types:**
- Entity/Asset references (drag-drop + dropdown)
- Sprite/Image (thumbnail preview)
- Material reference
- Layer mask (multi-select dropdown)

### 4.3 Toolbar Features

| Feature | Priority | Description |
|---------|----------|-------------|
| Bulk Add Rows | P0 | Add N rows with auto-naming ($i, $t variables) |
| Column Visibility | P0 | Show/hide columns, count indicator |

| Search | P0 | Query syntax compatible with Flecs |
| Pagination | P0 | 20 rows default, prevents expensive renders |
| Open/Save | P0 | Load/save to flat-file formats |
| Export | P1 | Export to CSV, YAML, JSON, HJSON |
| Paste Pad | P1 | Direct clipboard editing textarea |
| Schema Edit | P2 | MySQL-style table definition view |
| Entity Graph | P2 | Force-directed relationship visualization |

## 10. Acceptance Criteria

### 10.1 Functional Requirements

- [ ] User can create, read, update, and delete entity rows
- [ ] User can add/remove columns (schema modification)
- [ ] User can filter/search entities using query syntax
- [ ] User can copy/paste between editor and external spreadsheets
- [ ] User can import/export CSV, JSON, YAML files
- [ ] User can configure activity-based views
- [ ] User can navigate entity relationships via graph view
- [ ] Hot-reload triggers game engine updates within 100ms

### 10.2 Non-Functional Requirements

- [ ] Support 10,000+ entities without performance degradation
- [ ] Initial load time < 2 seconds
- [ ] Cell edit response time < 50ms
- [ ] File save operation < 500ms for typical project
- [ ] Zero data loss on crash (auto-save)
- [ ] Works in Chrome, Firefox, Safari, Edge

---

## 11. Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Performance issues at scale | Medium | High | Virtual scrolling, lazy loading, Web Workers |
| Complex nested data editing | High | Medium | Default to JSON view, optional flattening |
| Hot-reload latency | Low | High | Debounce changes, batch updates |
| Browser compatibility | Low | Medium | Test across browsers, use polyfills |
| Data corruption | Low | Critical | Validation on save, backup files |

## Appendix A: Glossary

| Term | Definition |
|------|------------|
| **Entity** | A game object with unique ID and component data |
| **Component** | A data container attached to entities (e.g., Health, Position) |
| **Asset** | A file on disk (sprite, audio, prefab) |
| **Prefab** | A reusable entity template |
| **Flecs** | An Entity Component System library |
| **Hot Reload** | Updating game data without restarting the game |
